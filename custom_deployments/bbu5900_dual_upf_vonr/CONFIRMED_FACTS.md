# BBU5900 Dual-UPF VoNR 项目已确认事实

更新时间：2026-04-15

本文档只记录在本项目推进过程中，已经通过配置核查、抓包、日志、现场拨测或基站侧查询明确确认的事实，不写未经验证的推测。

## 1. 项目范围与目标

- 当前工作目录为 `docker_open5gs/custom_deployments/bbu5900_dual_upf_vonr/`。
- 目标场景为：华为 BBU5900 商用基站 + 实体 UE 的 5G SA、VoNR、单 SMF + 双 UPF 组网。
- 现场当前关注的 APN 只有两个：`internet` 和 `ims`。

## 2. 架构与地址事实

以下架构已通过配置文件、运行状态和日志交叉确认：

- `internet` DNN 走 eUPF。
- `ims` DNN 走 Open5GS 原生 UPF。
- AMF N2 地址为 `10.10.10.100:38412`。
- eUPF N3 对外通告地址为 `10.10.10.101`。
- IMS UPF N3 经 `10.10.10.100:2152/udp` 暴露给 gNB。
- 现场物理接口职责确认如下：
  - `ens37`：N2 + IMS N3
  - `ens38`：internet N3
  - `ens39`：internet N6

## 3. 已完成并验证的修复

### 3.1 internet 数据面修复完成

已确认的问题与结果：

- eUPF 仅挂载单网卡时，internet 下行无法正常回封装。
- 将 XDP 接口改为 `ens38,ens39` 后，internet 业务恢复正常。
- UE 的 `internet` 地址池为 `10.45.0.0/16`，当前已可用。

### 3.2 宿主机多网卡回程与 ARP 漂移问题已处理

已确认的问题与结果：

- `ens37`、`ens38` 与 gNB 同 VLAN/同子网时，存在 ARP 与回程路径漂移。
- 通过 `arp_ignore=1`、`arp_announce=2` 和 UE 子网策略路由加固后，链路稳定性明显改善。

### 3.3 IMS 独立承载与 VoNR 基本能力已验证

已确认的问题与结果：

- `ims` 地址池为 `10.46.0.0/16`。
- 现场短号 `801003` 已完成 VoNR 成功呼叫验证。
- IMS 相关网元当前保持可用，未发现核心网侧明显崩溃或拒绝注册的问题。

## 4. gNB 与核心网标识匹配事实

通过 gNB 配置与核心网配置交叉核查，已确认以下关键标识一致：

- PLMN：`001/01`
- TAC：`1`
- AMF Name：`open5gs-amf0`
- GUAMI 可匹配当前 Open5GS 配置
- gNB 到 AMF 的 N2 SCTP 配置正确
- gNB 到双 UPF 的 N3 目标地址配置正确

## 5. 关于不活动释放与恢复的已确认事实

### 5.1 gNB 的释放原因已明确

通过 NG 接口跟踪已确认：

- UE 上下文释放由 gNB 先发起。
- NGAP 原因明确为 `radioNetwork / user-inactivity`。
- AMF 在该流程中是配合回收上下文，而不是主动异常踢线。

### 5.2 两类承载的超时与恢复时机本来就可以不同

通过基站侧参数表和现场现象已确认：

- 默认承载与 IMS 相关承载的不活动定时器不是同一个值。
- 因此两类承载不会天然要求“总是同时释放、同时恢复”。
- 在空闲态或恢复初期，先看到默认承载、随后再出现 IMS 专载，是可以被当前配置解释的。

### 5.3 空闲态下的业务恢复能力已经得到验证

通过最近一小时左右的抓包、核心网日志和基站侧查询，已确认：

- UE 被释放到空闲/Idle 后，只要有下行业务触发，能够重新接入。
- AMF 能看到 `InitialUEMessage` 与 `Service request`。
- 核心网最近检查窗口内未发现明确的 paging failure 证据。
- 基站侧已经观测到恢复后存在如下组合：
  - 仅默认承载
  - 默认承载 + IMS 专载
  - IMS 专载存在

因此，至少从“空闲态被下行业务重新拉回”这个角度看，系统主流程是通的。

## 6. IMS 用户面当前状态的已确认事实

通过在 IMS UPF 容器内部直接测试，已确认：

- 当前在线 UE 的 IMS 地址至少包含 `10.46.0.6` 与 `10.46.0.7`。
- 从 IMS UPF 容器内 ping 这两个地址可达。
- 从宿主机直接 ping 不通并不构成 IMS 路径故障，因为该地址池属于 IMS UPF 内部用户面路径。

这说明：

- IMS APN 的用户面当前不是“完全断的”。
- 问题重点不在 IMS 用户面本身不通，而在恢复触发时机与无线侧状态管理。

## 7. 基站侧承载观察的已确认事实

通过 `DSP GNBUEBASICINFO` 的结果，已确认近期现场可观察到：

- 存在同时带有两条 DRB 的 UE。
- 其中包括：
  - `服务质量等级 9` 的默认数据承载
  - `服务质量等级 5` 的 IMS 相关承载

这证明：

- 5QI/服务质量等级 5 的承载不是建不出来。
- 当前不是“IMS 专载绝对无法建立”的问题。

## 8. 第一阶段可交付结论

截至当前，第一阶段关于核心网侧的工作可以形成如下结论：

1. 单 SMF + 双 UPF 的核心架构已经跑通。
2. internet 与 ims 的分流逻辑已验证成立。
3. core network 没有发现导致当前现象的明确主故障。
4. 近期最关键的释放现象，根因已经确认主要来自 gNB 的 `user-inactivity` 策略。
5. 在空闲态下，只要有下行业务触发，UE 能够被重新拉回，系统主流程符合预期。
6. 当前阶段的主要矛盾已经不再是“核心网基本不可用”，而更偏向无线侧空闲管理与恢复策略的工程优化。

## 9. 后续阶段建议

后续若继续推进，建议将工作重点放在：

- gNB 侧 Inactive/Idle 切换策略优化
- 不同承载定时器与业务触发的协同验证
- 主叫/被叫场景下恢复时机的一致性验证
- 在不破坏当前成功基线的前提下做小步优化
