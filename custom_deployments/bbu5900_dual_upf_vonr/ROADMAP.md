# BBU5900 Dual-UPF VoNR 项目路线图

## 1. 项目定位

本项目的目标不是做一个面向大规模商用的运营商级平台，而是逐步打造一个适合 **个人或小团队内部使用** 的 5GC + IMS + 双 UPF 实验与验证环境，并持续演进到：

- 可稳定运行
- 可重复部署
- 可监控告警
- 可进行用户开通与鉴权维护
- 可支持基础计费与使用记录
- 可与外部 SIP 系统进行联调

当前阶段，核心主线仍然围绕：

1. 双 UPF 架构稳定化
2. VoNR 与 IMS 承载合理性验证
3. eUPF 数据面能力演进
4. 运维与平台化能力沉淀

---

## 2. 北极星目标

当本项目进入较成熟阶段时，应至少具备以下能力：

- 单 SMF + 双 UPF 的 DNN 级分流稳定可用
- internet 与 ims 承载分离明确，故障边界清晰
- 现场 UE 可稳定完成注册、上网、IMS 注册与 VoNR 呼叫
- 空闲态恢复链可复现、可度量、可审计
- 所有关键组件具备基础监控、告警与日志追踪能力
- 新环境可以按文档和脚本复现部署
- 用户开通、停复机、鉴权维护不再依赖手工改库
- 后续可进一步向 IBCF / 外部 SIP 对接扩展

---

## 3. 路线总览

| 阶段 | 主题 | 目标 | 当前建议 |
|------|------|------|----------|
| Phase 1 | 稳定基线 | 固化双 UPF + VoNR 原型，形成可重复验证的稳定版本 | 已进入收尾与固化阶段 |
| Phase 2 | 可观测性 | 建立 Prometheus / Grafana / 告警 / 日志审计体系 | 建议立即开始 |
| Phase 3 | 数据面演进 | 验证 eUPF 原生模式与性能收益 | 待监控基线稳定后开展 |
| Phase 4 | 自动化运维 | 统一部署、备份、回退、巡检与变更流程 | 中期重点 |
| Phase 5 | 用户与计费平台 | 统一开户、鉴权与基础计费管理 | 后续独立推进 |
| Phase 6 | SIP 外联 | 实现 VoNR 与外部 SIP 系统互通 | 平台成熟后推进 |

---

## 4. 各阶段细化

### Phase 1：稳定现有双 UPF 与 VoNR 原型

**目标**
- 固化当前稳定版本
- 建立 UE 兼容性矩阵
- 保持 internet 与 ims 分流正确
- 继续验证空闲态恢复链与 VoNR 呼叫链路

**重点工作**
- 扩展验收清单，形成稳定版本基线
- 针对不同品牌或芯片平台 UE 建立测试矩阵
- 将现有脚本纳入固定回归流程
- 在每次重大改动前后保留日志与抓包证据

**完成标志**
- 形成稳定性基线文档
- 已知问题与风险点有明确记录
- 主要业务链路具备复现证据

### Phase 2：建立最小可用监测、告警、仪表盘

**目标**
- 让系统从“靠人工看日志”升级为“可观测、可告警、可追踪”

**重点工作**
- 补齐网元健康、PFCP 状态、双 UPF 分流、IMS 注册与呼叫健康面板
- 将宿主机 CPU、内存、网卡流量和丢包纳入观测范围
- 建立最小告警集（P0 ~ P2）
- 形成日志审计与故障时间线能力

**完成标志**
- 出现关键故障时可快速判断范围
- 基本告警机制可用
- 日志与指标能相互印证

### Phase 3：推进 eUPF 数据面演进

**目标**
- 在稳定和可观测的前提下，验证 eUPF 从当前模式向更高性能模式演进的可行性

**重点工作**
- 检查内核、驱动、XDP attach 模式支持情况
- 在小窗口中进行单变量切换验证
- 对比时延、吞吐、CPU、错误日志与断流表现
- 形成明确的回退策略

**完成标志**
- 拿到可靠的性能对比结果
- 明确是否值得将模式切换纳入长期默认方案

### Phase 4：把部署与变更流程产品化

**目标**
- 让部署、升级、回退、巡检形成标准化流程，降低人为操作风险

**重点工作**
- 整理当前宿主机初始化、策略路由、部署脚本
- 统一环境模板、回退步骤、备份恢复方式
- 引入 Ansible 等轻量自动化工具
- 固化 Runbook 与值守手册

**完成标志**
- 新环境可以按文档和脚本快速复现
- 关键操作可回退、可审计

### Phase 5：建立统一开户、鉴权与基础计费能力

**目标**
- 从“手工维护用户数据”演进到“有主数据、有流程、有记录”的内部管理方式

**重点工作**
- 梳理 5GC 与 IMS 两套用户与鉴权数据模型
- 明确主数据归属与同步方向
- 先做最小内部管理入口，再逐步扩展
- 引入基础使用记录与离线计费能力

**完成标志**
- 用户开通、停复机、鉴权维护具备统一入口
- 人工直改数据库的频率明显下降

### Phase 6：推进 VoNR 与外部 SIP 系统对接

**目标**
- 在内部平台稳定后，实现与外部 SIP 系统的可控互通

**重点工作**
- 参考 IBCF 目录与现有 IMS 组件推进联调
- 规范号码、路由与安全边界
- 完成基础 SIP 互通与故障排查手册

**完成标志**
- 完成至少一条稳定的 SIP 外联验证链路
- 具备后续扩展能力

## 5. 仓库与协作策略

当前建议采用 **一主两辅** 的仓库结构：

- 主仓库：当前 `docker_open5gs` 定制部署仓库
- 参考仓库：`eupf-main`
- 参考仓库：`open5gs-main`

现阶段 **不建议** 单独新建“项目进度仓库”或“监控仓库”，原因是：

- 会打散代码、配置、文档与运行上下文
- 增加后续协作与维护成本
- 降低问题定位效率

当前更适合的方式是：

- 在主仓库中持续维护状态文档、确认事实文档、验收清单与路线图
- 配合 GitHub 提交记录和里程碑进行进度追踪
- 等进入用户/计费平台阶段时，再单独建立新的内部平台仓库

---

## 6. 推荐工具栈

### 当前阶段优先推荐
- 部署：Docker Compose
- 自动化：Ansible
- 指标监控：Prometheus
- 仪表板：Grafana
- 告警：Alertmanager
- 日志：先保留当前脚本化审计，后续可按需补 Loki / Promtail
- 统一平台：FastAPI 或 Django + PostgreSQL

### 暂不建议过早投入
- 过重的微服务拆分
- 复杂的在线计费系统
- 为监控或进度管理单独再拆多个仓库
- 过早切换到更重的编排系统

---

## 7. 进度追踪原则

后续建议始终遵循以下原则：

1. **事实与判断分离**：确认过的事实写入 `CONFIRMED_FACTS.md`
2. **状态持续更新**：阶段进展写入 `PROJECT_STATUS.md`
3. **验收口径固定**：所有重要功能都要能回到验收清单中验证
4. **重大变更留痕**：保留版本号、日志样本、抓包样本与回退方案
5. **先稳定、再扩展**：优先稳住主链路，再做模式切换与平台开发

---

## 8. 当前建议的下一步

如果继续按照本路线推进，建议下一阶段直接从以下三件事开始：

1. 把阶段 2 的监控与告警体系正式补齐
2. 建立一份持续使用的 UE 兼容性矩阵
3. 为 eUPF 原生模式切换准备验证基线与回退方案

这三项完成后，项目就会从“现场验证型”明显进入“可持续演进型”。
